信息架构与关键流程

三栏工作台

  • 左栏 · 会话：按工作目录自动分组（与 ~/.claude/projects/ 的组织方式同构），支持置顶、状态点（运行 / 待确认 / 空闲）、⌘K 全局搜索。
  • 中栏 · 会话流：消息 + 工具调用折叠行 + diff 块 + 沉淀建议事件，底部常驻输入框（可换模型 / 权限模式）。
  • 右栏 · .claude 上下文：把「这轮到底加载了什么」显性化——指令层级、唤起的记忆、可用 Skills、改动文件。这是与纯 CLI 最大的体验差异点。

记忆沉淀流程

  1. 会话中模型触发沉淀建议（内联事件卡，可当场确认）；
  2. 未处理的进入收件箱暂存，不写盘；
  3. 用户确认后才写 memory/<slug>.md 并更新 MEMORY.md 索引；
  4. 「整理记忆」= 手动触发合并去重（consolidate）。
  • 与你现有协议一致：确认前绝不落盘，宁缺毋滥。

实现前待对齐的决策分支（grill-me 清单）

这版稿只钉「长什么样」，下面每支都会改变「怎么建」，实现前需逐个钉死：

  • ① 壳与栈：Tauri（轻、原生感好）vs Electron（生态熟、坑少）。
  • ② 会话引擎（最重的一支）：嵌 Claude Agent SDK、把 base URL 指向 LiteLLM 网关（工具调用/权限/子代理白拿，但非 Claude 系模型的兼容性受 LiteLLM 转换质量制约）vs 自研 agent loop（全模型可控，但工具执行、权限门、回放要全部自建）。
  • ③ .claude 互通深度：transcript 与 CLI 完全同构（同一会话可在 CLI 续跑，最强但被上游格式绑死）vs 只共享指令与记忆、会话记录自有格式。
  • ④ 多模型记忆口径：非 Claude 模型产出的沉淀建议是否走同一确认协议、同一 frontmatter 规范。
  • ⑤ 权限模型：自动执行 / 计划先行 / 每步确认三档是否够，危险命令（删除、部署、远程写）是否单独硬门。

本稿为静态高保真原型，界面数据均为示意；确认方向后按此稿拆组件与状态。
